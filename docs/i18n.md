## Translation Workflow

### 1. UI Strings (Dictionary)

**Location:** `src/i18n/locales/{locale}.ts`

**Process:**
1. Edit `src/i18n/locales/en.ts` (canonical)
2. Copy changes to all 5 locale files
3. Translate each string
4. Run `pnpm build` — TypeScript verifies completeness

**String Types:**
- **Simple:** `title: "Blog"`
- **Function:** `minuteRead: (m) => \`${m} min read\``
- **Nested:** `pages.blog.title: "Blog"`

### 2. Content (MDX)

**Location:** `src/content/{collection}/{slug}/{locale}.mdx`

**Process:**
1. Create entry folder: `src/content/blog/my-post/`
2. Create `en.mdx` with full frontmatter
3. Create locale variants: `pt-br.mdx`, `ja.mdx`, etc.
4. Translate frontmatter + body
5. Set `canonicalSourceLocale: en` on translations

## Adding a New Locale

### Step-by-Step

#### 1. Update Locale Config (`src/i18n/config.ts`)

```typescript
export const locales = [
  "en", "pt-br", "ja", "zh-cn", "de", "fr", "es"  // add "es"
] as const;
```

#### 2. Add Metadata (`src/i18n/config.ts`)

```typescript
export const localeMetadata: Record<Locale, { label: string; languageTag: string }> = {
  // ... existing
  es: { label: "Español", languageTag: "es" },
};
```

#### 3. Create Dictionary (`src/i18n/locales/es.ts`)

Copy `en.ts`, translate all strings:

```typescript
import type { Dictionary } from "@/i18n/types";

export const es: Dictionary = {
  meta: { siteDescription: "..." },
  navigation: { blog: "Blog", snippets: "Fragmentos", ... },
  common: { ... },
  projectStatuses: { ... },
  home: { ... },
  pages: { ... },
} satisfies Dictionary;
```

#### 4. Register Dictionary (`src/i18n/index.ts`)

```typescript
import { es } from "@/i18n/locales/es";

const dictionaries: Record<Locale, Dictionary> = {
  en,
  "pt-br": ptBr,
  ja,
  "zh-cn": zhCn,
  de,
  fr,
  es,  // add
};
```

#### 5. Update Astro Config (`astro.config.ts`)

```typescript
i18n: {
  locales: ["en", "pt-br", "ja", "zh-cn", "de", "fr", "es"],
  defaultLocale: "en",
  routing: { prefixDefaultLocale: false },
}
```

#### 6. Update Sitemap Config (`astro.config.ts`)

```typescript
sitemap({
  i18n: {
    defaultLocale: "en",
    locales: {
      // ... existing
      es: "es",
    },
  },
})
```

#### 7. Scaffold Content

```bash
pnpm content:new:post   # creates es.mdx for all posts
pnpm content:new:snippet
```

#### 8. Verify

```bash
pnpm build
# Should complete with new locale pages
```

## Content Translation Strategies

### Strategy 1: Human Translation (Recommended)

- Create all 6 locale files manually
- Set `translation.generator: "human"`
- Best quality, full control

### Strategy 2: Machine Translation + Review

```bash
# 1. Set up translation adapter (CLI tool)
export CONTENT_TRANSLATION_COMMAND="npx my-translator-cli"
export CONTENT_TRANSLATION_ARGS='["--model", "gpt-4o"]'

# 2. Run translation
pnpm content:translate

# 3. Review generated files
# 4. Set translation.generator appropriately
```

**Adapter Interface (stdin/stdout):**

```json
// Input (stdin)
{
  "collection": "blog",
  "slug": "my-post",
  "sourceLocale": "en",
  "targetLocale": "es",
  "sourcePath": "src/content/blog/my-post/en.mdx",
  "sourceContent": "---\ntitle: Hello\nlocale: en\n---\nBody..."
}

// Output (stdout) — MUST include locale in frontmatter
---
title: Hola
locale: es
canonicalSourceLocale: en
translation:
  sourceLocale: en
  generatedAt: "2025-01-15T14:30:00Z"
  generator: "gpt-4o"
---
Cuerpo...
```

### Strategy 3: Hybrid

- Core UI: Human (dictionary)
- Content: Machine + human review
- Set `generator` to track method

## Managing Missing Translations

### UI Strings

- **Never missing** — TypeScript enforces all keys in all locales
- Build fails if any locale incomplete

### Content

- **Fallback to EN** — `getLocalizedListing()` shows EN with `isFallback: true`
- **Language Switcher** — Unavailable locales shown disabled
- **Content Page** — Shows "Available in English" banner

### Finding Missing Translations

```bash
pnpm content:check
```

Output:
```
Missing translations:
- blog/my-post/es.mdx
- snippets/docker-tips/es.mdx
- projects/my-tool/es.mdx
```

## Translation Quality Checklist

### Per String

- [ ] Meaning preserved (not literal)
- [ ] Tone matches brand (technical, personal, precise)
- [ ] Terminology consistent with existing translations
- [ ] Pluralization handled (use `minuteRead` function)
- [ ] No English leakage (UI labels, alt text, aria-labels)

### Per Content Page

- [ ] Frontmatter translated (title, description, tags)
- [ ] Body fully translated
- [ ] Code comments translated (if applicable)
- [ ] Links updated to localized versions
- [ ] `canonicalSourceLocale` set
- [ ] `translation.generator` accurate

### Per Locale

- [ ] All dictionary keys present
- [ ] All content entries have locale variant OR accepted fallback
- [ ] Language switcher shows locale
- [ ] hreflang tags correct
- [ ] Date/number formats appropriate

## Locale-Specific Notes

### Portuguese (pt-BR)

- **Formality:** Informal "você" (not "tu" or "vós")
- **Tech terms:** Often keep English (API, CLI, deploy, cache)
- **Date:** DD/MM/YYYY
- **Pluralization:** `minuteRead(n)` → `n min de leitura`

### Japanese (ja)

- **Script:** Kanji + Hiragana + Katakana
- **Tech terms:** Katakana (API → エーピーアイ)
- **No spaces** between words
- **Date:** YYYY年MM月DD日
- **Reading time:** Character-based (~400 chars/min)
- **Pluralization:** No grammatical plural

### Chinese Simplified (zh-cn)

- **Script:** Simplified Hanzi
- **Tech terms:** Often translated (API → 接口, deploy → 部署)
- **No spaces** between words
- **Date:** YYYY年MM月DD日
- **Reading time:** Character-based (~400 chars/min)
- **Pluralization:** No grammatical plural

### German (de)

- **Formality:** "Sie" form (not "du")
- **Compounds:** Long compound words (Datenschutzerklärung)
- **Date:** DD.MM.YYYY
- **Pluralization:** Complex — use `minuteRead(n)` function

### French (fr)

- **Formality:** "Vous" form
- **Spaces:** Before `: ; ? ! »` (use `&nbsp;`)
- **Date:** DD/MM/YYYY
- **Pluralization:** `minuteRead(n)` → `n min de lecture`

## Updating Translations

### UI Strings

1. Modify `en.ts`
2. Update all 5 locale files
3. `pnpm build` — CI catches missing

### Content

1. Update `en.mdx` (canonical)
2. Update `updatedDate` in EN
3. Update translations:
   - Update body
   - Update `updatedDate` in translation
   - Update `translation.generatedAt`
4. Or: re-run `pnpm content:translate` for that entry

### Batch Updates

For large changes (rebranding, terminology):

1. Script to update all locale files
2. Or: Use translation memory tool (TMX, XLIFF)
3. Re-validate with `pnpm check`

## Common Pitfalls

| Pitfall | Solution |
|---------|----------|
| Missing locale in `locales` array | Add to `config.ts` + `astro.config.ts` + `index.ts` |
| `locale` frontmatter ≠ filename | Must match exactly (`pt-br.mdx` → `locale: pt-br`) |
| Forgot `canonicalSourceLocale` | Add to translation frontmatter |
| English in translated UI | Check all 6 locale files |
| Date format wrong | Use `publishedDate` (ISO) — formatting in components |
| Pluralization broken | Use `dict.common.minuteRead(n)` function |
| Language switcher broken | Check `getLocalizedPath()` logic |

## Migration from Other i18n Systems

### From JSON/i18next

1. Export keys → TypeScript object
2. Create `src/i18n/locales/{locale}.ts`
3. Remove runtime i18n library
4. Update components to use `getDictionary()`

### From Astro i18n (built-in)

This project uses **custom dictionaries** instead of Astro's built-in i18n routing for:
- Full TypeScript types
- Function-based pluralization
- No runtime overhead
- Content fallback control

Astro's `i18n` config used only for:
- Routing (`prefixDefaultLocale: false`)
- Sitemap `hreflang` generation
- `<html lang>` via `localeMetadata`
